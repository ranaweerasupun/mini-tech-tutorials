# 05 — I2C Protocol and the Kernel I2C Subsystem

You now understand how the kernel matches a driver to a device and what happens when that match occurs. The next piece to understand is the actual communication protocol your driver will use to talk to the TMP102. I2C is not complicated, but it has a specific vocabulary and a specific mental model, and you need both before the driver code will feel natural.

---

## What I2C Is

I2C (Inter-Integrated Circuit, pronounced "I squared C") is a two-wire serial communication protocol invented by Philips in the 1980s for connecting integrated circuits to each other on the same circuit board. Despite being decades old, it remains one of the most widely used protocols in embedded systems because it is simple, cheap, and requires only two wires regardless of how many devices are connected.

Those two wires are SDA (Serial Data) and SCL (Serial Clock). SCL carries a clock signal generated by the bus master — in your case, the Raspberry Pi. SDA carries data in both directions, one bit at a time, synchronised to the clock. Every bit of data is sent by holding SDA high or low while SCL pulses, and every bit is read by sampling SDA when SCL is high. The protocol defines start and stop conditions, acknowledgement bits, and a specific frame format, but the kernel's I2C subsystem handles all of that for you. You never manipulate the clock or data lines directly.

The key architectural property of I2C is that multiple devices can share the same two wires simultaneously. Each device has a unique 7-bit address, and every transaction begins with the master broadcasting an address. Only the device with the matching address responds. The TMP102's address is determined by the ADD0 pin: when ADD0 is connected to ground (as in our wiring), the address is 0x48. When ADD0 is connected to VCC, the address is 0x49. This address-selection mechanism allows up to four TMP102 sensors on the same I2C bus, each with a different address.

---

## Registers: How You Ask the Sensor for Data

Almost every I2C sensor organises its interface as a set of registers — small memory locations inside the sensor chip, each with a specific address. You interact with the sensor by reading and writing these registers. The TMP102 has four registers, but only one matters for this tutorial: register 0x00, the temperature register, which holds the most recently measured temperature value.

Reading a register over I2C is a two-step transaction. First, you send a write message to the sensor containing the register address you want to read (this tells the sensor "I want the contents of this particular register"). Then immediately after, without releasing the bus, you send a read message and receive the register's contents back. This combined write-then-read pattern is called a register read, and it is by far the most common operation in I2C driver code.

The TMP102 temperature register is 16 bits wide (two bytes). When you read it, you receive two bytes back. The temperature is encoded in the upper 12 bits of those 16 bits, with the lower 4 bits unused. Each unit of the 12-bit value represents 0.0625°C — so to convert to millidegrees Celsius, you shift the raw value right by 4 bits and multiply by 62.5, which in integer arithmetic becomes multiplying by 1000 and dividing by 16.

---

## SMBus: The Simplified Subset Your Driver Uses

The kernel's I2C API has two levels. The lower level, the raw I2C API, gives you complete control over individual messages — you construct each start condition, address byte, data byte, and stop condition explicitly. This is necessary for some complex protocols but is verbose for the common cases.

The higher level is SMBus, a subset of I2C defined by Intel that codifies the most common patterns — reading a byte register, reading a word register, writing a byte, and so on — into simple function calls. SMBus is not a different protocol on the wire; it is just a more convenient API for the common cases. The TMP102 follows SMBus conventions, so your driver uses the SMBus functions exclusively.

The one SMBus function the TMP102 driver uses is `i2c_smbus_read_word_swapped`:

```c
s32 raw = i2c_smbus_read_word_swapped(client, TMP102_TEMP_REG);
```

This single function call does everything: it sends the register address (0x00) to the TMP102, reads back two bytes of data, and swaps them into the correct byte order. The byte swap is necessary because the TMP102 sends its bytes in big-endian order (most significant byte first), but x86 and ARM processors are little-endian (least significant byte first). The `_swapped` variant handles the conversion automatically.

The function returns an `s32` (signed 32-bit integer). If the read succeeded, the lower 16 bits contain the register value. If it failed — because the sensor did not respond, or there was a bus error — it returns a negative error code. This is the standard kernel pattern: check for negative return values to detect errors.

## The i2c_client Struct: Your Connection to the Hardware

Throughout this tutorial you have seen the type `struct i2c_client *client` passed to probe, remove, and the temperature reading function. It is worth taking a moment to understand what this struct represents, because it is the central object in every I2C driver.

An `i2c_client` represents one specific device on one specific I2C bus. The kernel creates it when it processes the device tree node for your sensor and populates it with the information from that node: the bus number, the device address (0x48 in our case), the device name, and a pointer back to the parent bus adapter. Every SMBus function takes an `i2c_client` as its first argument, because the function needs to know which bus to use and which address to address.

When you store `data->client = client` in your probe function, you are saving this connection object so that other parts of the driver (like the temperature reading function, which is called from the sysfs attribute rather than receiving the client directly) can use it. Without storing the client pointer in your private data struct, you would have no way to communicate with the hardware from callback functions.

## Testing I2C Access From the Command Line

Before loading any driver, you can interact directly with I2C devices from the command line using `i2c-tools`. This is an invaluable diagnostic capability — it lets you verify the hardware is working and understand what a raw register read looks like, independently of your driver code.

First, confirm the sensor is on the bus:

```bash
sudo i2cdetect -y 1
```

The `-y 1` means "scan I2C bus 1 without asking for confirmation." Look for `48` in the output grid — that is your TMP102 responding. If you see `UU` instead of `48`, it means a driver has already claimed the device (the in-kernel TMP102 driver may be loaded). If you see only `--`, the device is not responding — double-check your wiring.

You can also read a register directly:

```bash
# Read a 16-bit word from register 0x00 of the device at address 0x48 on bus 1
sudo i2cget -y 1 0x48 0x00 w
```

The `w` at the end means read a word (16 bits). The value you get back should look like `0xd817` or something similar (the exact value depends on your room temperature). This is the raw register value that your driver will read, decode, and present as a temperature. Being able to check this value with `i2cget` and compare it against your driver's calculation is a useful sanity check when you first get the driver running.

---

## What the Kernel's I2C Subsystem Does For You

One last thing worth appreciating before moving to the hwmon subsystem: the amount of work the kernel's I2C subsystem is doing transparently. When you call `i2c_smbus_read_word_swapped`, the kernel is managing the I2C bus controller hardware (which on the Pi is a peripheral of the BCM2711 SoC), handling the electrical signalling via the I2C adapter driver (which is a separate kernel driver specifically for the Pi's hardware), performing the multi-byte transaction with proper start/stop conditions and acknowledgement handling, managing bus arbitration if multiple masters are present, and returning the result to your code.

Your driver never sees any of this. It calls one function with a register address and gets back the data. The entire protocol stack between your call and the physical SDA and SCL pins is handled by layers of the kernel that you do not need to understand to write your driver. This is the power of layered architecture — your driver speaks the I2C API, and the API speaks to the hardware.
